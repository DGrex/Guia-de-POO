Asociación Agregación Composición Generalización
💡
UML en el mundo real: El Lenguaje Unificado de Modelado (UML) define cómo interactúan los objetos entre sí. Aunque en Python todas las variables guardan referencias a posiciones de memoria, la forma en que los objetos se instancian, inyectan o destruyen define conceptualmente el tipo de acoplamiento del sistema.
Teoría

Tipos de Acoplamiento y Dependencia

1. Asociación (Relación "Usa un")

Concepto: Es la relación estructural más débil. Una clase interactúa con otra pasándola como argumento o asignándola a una propiedad, pero ambas mantienen ciclos de vida completamente independientes y autónomos.

asociacion.py
class Profesor:
    def __init__(self, nombre):
        self.nombre = nombre

class Curso:
    def __init__(self, titulo, profesor):
        self.titulo = titulo
        self.profesor = profesor  # El profesor existe fuera del curso

p = Profesor("Ana")
c = Curso("Matemáticas", p)
print(c.profesor.nombre)  # Salida: Ana

2. Agregación (Relación "Tiene un" / Todo-Parte Débil)

Concepto: Una clase actúa como contenedora ("todo") de una colección de objetos independientes ("partes"). Si el contenedor principal se destruye, los objetos que estaban almacenados en su interior siguen existiendo en memoria.

agregacion.py
class Estudiante:
    def __init__(self, nombre):
        self.nombre = nombre

class Escuela:
    def __init__(self, nombre):
        self.nombre = nombre
        self.estudiantes = []  # Los estudiantes se agregan externamente

    def agregar_estudiante(self, estudiante):
        self.estudiantes.append(estudiante)

e1 = Estudiante("Luis")
escuela = Escuela("Colegio Central")
escuela.agregar_estudiante(e1)
print(escuela.estudiantes[0].nombre)  # Salida: Luis

3. Composición (Relación "Formado por" / Todo-Parte Fuerte)

Concepto: Es un acoplamiento estricto. La clase contenedora se encarga de instanciar directamente a los objetos internos. El ciclo de vida de las partes está supeditado al del contenedor; si este se destruye, sus partes mueren con él.

4. Herencia UML (Relación "Es un" / Generalización)

Concepto: Una relación de especialización donde una subclase se deriva de una superclase, heredando toda su interfaz y estado estructural para extender o redefinir comportamientos específicos.

herencia_uml.py
class Persona:
    def __init__(self, nombre):
        self.nombre = nombre

class Estudiante(Persona):
    def __init__(self, nombre, carrera):
        super().__init__(nombre)  # Reutiliza el constructor de la superclase
        self.carrera = carrera

e = Estudiante("Carlos", "Ingeniería")
print(e.nombre, e.carrera)  # Salida: Carlos Ingeniería

Ejercicio 2: Colección estructural de entidades volátiles (Agregación)

Modela una división corporativa capaz de almacenar y listar personal operativo inyectado dinámicamente desde el exterior del objeto contenedor.

class Empleado:
    def __init__(self, nombre):
        self.nombre = nombre

class Departamento:
    def __init__(self, nombre):
        self.nombre = nombre
        self.empleados = []

    def agregar_empleado(self, empleado):
        self.empleados.append(empleado)

d = Departamento("TI")
d.agregar_empleado(Empleado("Ana"))
print(d.empleados[0].nombre)  # Salida: Ana

Ejercicio 3: Instanciación e internalización rígida de componentes (Composición)

Construye una estructura arquitectónica habitacional donde las dependencias internas se inicializan de forma privada y dependiente en el constructor de la raíz.

casa_habitaciones.py 
class Habitacion:
    def __init__(self, nombre):
        self.nombre = nombre

class Casa:
    def __init__(self):
        # Las habitaciones nacen y mueren única y estrictamente con la Casa
        self.habitaciones = [Habitacion("Sala"), Habitacion("Cocina")]

c = Casa()
for h in c.habitaciones:
    print(h.nombre)
# Salida:
# Sala
# Cocina
